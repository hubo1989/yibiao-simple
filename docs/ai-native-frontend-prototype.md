# 易标 AI Native 前端原型需求梳理

## 1. 产品定位

易标不是单点写作工具，而是“投标标书智能体工作台”。核心价值是把招标文件、企业素材、响应矩阵、章节正文、质量审查和交付导出组织成一条可追溯、可审核、可协作的项目级工作流。

## 2. 实际功能面

| 功能域 | 当前系统证据 | 前端应承载的用户任务 |
| --- | --- | --- |
| 项目工作台 | `/api/projects`, dashboard, progress | 查看项目状态、风险、最近动作，快速进入当前交付任务 |
| 招标文件解析 | `/api/document/upload-to-project`, `/analyze-project-stream` | 上传招标文件，流式查看解析结果，确认项目概况和技术要求 |
| 目录与章节生成 | `/api/outline/*`, `/api/content/*` | 生成三级目录，按章节生成正文，人工编辑、锁定、评论和版本回滚 |
| 响应矩阵 | `/api/response-matrix/*`, scoring | 逐条绑定招标条款、评分项、响应章节和证据来源 |
| 废标/评分/一致性检查 | disqualification, scoring, consistency | 把风险前置成质量门禁，而不是导出前才发现问题 |
| 标书审查 | `/api/review/*` + PRD | 上传投标文件，按响应性/合规性/一致性审查，导出批注 Word |
| 知识库与素材库 | knowledge, materials, LlamaIndex | 管理历史标书、企业资料、证照、案例，并匹配到章节或附件需求 |
| 模板与交付 | templates, export, export-template | 套用 Word 模板，执行导出前检查，生成正式交付包 |
| 协作治理 | members, comments, versions, request logs, admin | 管理成员权限、批注、版本、提示词、模型 Key 和审计日志 |
| 项目级智能体 | `/api/bid-agent/*` | 一键生成技术标初稿、一键修复风险，并可查看 agent 执行步骤 |

## 3. AI Native 设计原则

1. **Command-first**：顶部常驻指令栏，让用户以“生成初稿 / 修复风险 / 匹配素材 / 导出交付包”发起任务。
2. **Single entry**：全局层只保留一个主聊天入口，承接“上传招标文件生成项目”和“查找已有项目继续管理”。
3. **Serial project workflow**：进入项目后按标书生产依赖串行展示步骤，避免用户在菜单树里寻找功能。
4. **Jump when data exists**：已存在数据的步骤可任意跳转、复核、重跑；缺前置条件的步骤展示门禁原因。
5. **Evidence by default**：正文和审查结论默认显示证据来源、引用位置、置信度和待确认状态。
6. **Human approval gates**：AI 结果进入交付前必须经过人工确认，尤其是证照、资质、承诺条款和 Word 导出。

## 4. 页面原型范围

本轮前端效果原型新增独立路由：

- `/ai-native-prototype`
- `/prototype`

原型收敛为两个关键层级：

| 层级 | 目标 |
| --- | --- |
| 全局单入口 | 通过聊天框上传招标文件生成新项目、搜索已有项目、继续待办项目 |
| 项目串行流程 | 按“上传招标文件 → 解析确认 → 响应矩阵 → 目录章节 → 内容生成 → 质量审查 → 导出交付”推进 |

## 5. 后续落地建议

1. 将全局首页收敛为聊天框 + 项目搜索结果，弱化传统菜单。
2. 将项目详情页改造成串行步骤流，并用状态标识“已完成 / 当前 / 可执行 / 风险 / 门禁未开放”。
3. 将 `/api/response-matrix/*` 和 `/api/bid-agent/*` 作为项目流程中的调度能力，而不是单独暴露成复杂入口。
4. 把导出前检查变成固定 quality gate，覆盖废标项、评分覆盖率、素材缺失、批注未处理和模板配置。
